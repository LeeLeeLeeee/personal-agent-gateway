실행 lifecycle과 취소

Job queue와 Team Hook 연동 상태의 실제 ownership을 검토한다.

60초 요약

JobWorker는 consumer 하나지만 job_worker_concurrency 설정은 2 이상도 수용·노출한다. Team Hook은 Hook Run과 Cycle 상태를 서로 다른 서비스가 갱신해 link 이후 중간 실패에서 상태가 갈릴 수 있다. 실행기 전체 재설계 없이 계약 축소와 로컬 transition coordinator를 비교한다.

사용자·도메인 행동

B-01

queued Job이 단일 JobWorker Queue를 거쳐 Runner에서 terminal 상태가 된다.

B-02/B-03

HookRun은 Cycle에 연결되어 projection과 Team Runtime을 거치고 restart 시 각 서비스가 active 상태를 복구한다.

현재 구조

결정 질문: Team Hook 중간 실패의 상태 owner는 누구인가?

HookRunner → TeamRunService.create_cycle
           → HookRunService.link_cycle
           → MailProjector
failure:   → HookRunService.mark_interrupted
             Team Cycle remains independently stored
success:   → TeamOrchestrator.run_cycle → settle observer

실제 호출 순서를 표현한 텍스트 fallback이다.

F-01 worker concurrency

OptionBenefitCostRiskApplicable when
O-01/A 유지변경 없음effective=1 설명설정 불신예약 필드
O-01/B 값 1동작 정합validation향후 재확장병렬 근거 없음

F-02 HookRun/Cycle 전이

OptionBenefitCostRiskApplicable when
O-02/A 유지서비스 독립성 유지불일치 운영 절차active/terminal 상태 분리중간 실패가 허용됨
