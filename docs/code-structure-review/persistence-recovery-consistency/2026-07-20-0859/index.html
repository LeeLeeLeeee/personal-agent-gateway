저장·복구 일관성

SQLite backup과 파일 저장소의 실제 복구 경계를 검토한다.

60초 요약

현재 Backup은 의도적으로 SQLite만 복원하고 Auth, Transcript, Artifact, Workspace는 metadata만 기록한다. 이 계약은 문서와 테스트에 명시되어 있다. 그러나 Hook credential은 inventory에도 없어 DB restore 후 누락을 dry-run에서 찾을 수 없다.

사용자·도메인 행동

B-01/B-02

SQLite online backup을 만들고 checksum, schema, integrity를 검증해 별도 target으로 복원한다.

B-03

Hook DB row는 connection_ref를 보관하고 credential 본문은 별도 JSON 파일에 둔다.

현재 구조

결정 질문: DB backup 성공 시 어떤 상태가 함께 복구되지 않는가?

BackupService → SQLite copy
             → metadata manifest - - Auth / Transcript / Artifact / Workspace
Hook secret files - - no manifest edge - - BackupService
restore_to → SQLite copy only

텍스트 fallback이며 점선은 metadata-only 또는 누락 edge다.

F-01 복구 단위

OptionBenefitCostRiskApplicable when
O-01/A DB 유지작고 검증됨제한 설명전체 복구 오해DB metadata가 목적
O-01/B profile 분리복구 기대 명확manifest/UI 변경profile만 늘 수 있음단계적 복구
O-01/C full bundle전체 상태 보존암호화·용량 정책secret 유출RPO/RTO 확정

Recommendation: O-01/B. DB backup은 유지하고 database-only profile과 누락 저장소를 표시한다. DB만 필요하다는 사용성 근거가 있으면 O-01/A를 유지한다.
