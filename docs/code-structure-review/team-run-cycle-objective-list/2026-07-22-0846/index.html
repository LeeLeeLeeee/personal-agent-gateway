Feature / Module Review

Team Run 목표와 Cycle 중심 목록 구조 검토

Team Run 생성 계약, Cycle objective 전달, Team Runs list read model과 카드 정보 계층을 함께 검토한다.

60초 요약

현재 goal은 Run 제목, AUTO 반복 기준, Agent 공통 프롬프트를 동시에 담당한다. TRIGGERED는 별도 Cycle instruction도 받아 의미가 중복된다. 목록은 전체 Task를 합산해 Cycle이 늘수록 현재 실행 상태를 파악하기 어렵다.

사용자·도메인 행동

B-01 · Run 생성

Team과 AUTO/TRIGGERED 정책을 골라 Continuous container를 만든다.

B-02 · Cycle 실행

AUTO는 반복 기준으로, Trigger/Hook은 요청 instruction으로 Cycle을 실행한다.

B-03 · 목록 탐색

현재 Cycle 상태와 다음 행동을 확인하고 상세로 이동한다.

현재 구조

결정 질문: 현재 goal과 instruction은 실제 실행에서 어떻게 합쳐지는가?

flowchart LR
  Form[TeamPicker goal] --> Run[team_runs.goal]
  Trigger[Trigger or Hook instruction] --> Request[team_cycle_requests.instruction]
  Auto[AUTO scheduler] --> Fixed[Continue the team goal]
  Fixed --> Request
  Request --> AddWork[add_work task planning]
  Run --> AddWork
  AddWork --> Tasks[Cycle tasks]
  Tasks --> Execute[worker execution]
  Run --> Execute

  Synthesize --> Cycle[TeamRunCycle result]

Unknowns

  • TRIGGERED Run에 선택적인 장기 context가 필요한지는 확정되지 않았다.
  • 개별 Cycle 감사·검색이 목록의 주 사용 사례인지 실제 사용 데이터가 없다.

Recommendation

O-01/B + O-02/B: AUTO에만 base objective를 필수로 두고 모든 Cycle instruction을 실제 objective로 사용한다. 목록은 Run당 카드 하나를 유지하면서 Team 이름, 최신 Cycle objective/status, 최신 Cycle Task 진행률, Cycle 수와 AUTO 진행 정보를 표시한다.

신규 Run은 기존 rules_snapshot.team.name과 short Run ID를 함께 제목으로 사용하고, snapshot이 없는 legacy Run은 short Run ID만 사용한다. 별도 사용자 입력 Run 이름은 추가하지 않는다.

Reversal conditions

  • 모든 Triggered Cycle이 하나의 장기 목표 아래에만 존재한다는 규칙이 확정되면 O-01/A를 선택한다.
  • AUTO도 매회 외부 입력을 받게 되면 O-01/C를 다시 검토한다.
  • 개별 Cycle 감사가 주 사용 사례가 되면 O-02/C를 별도 History 화면으로 도입한다.
