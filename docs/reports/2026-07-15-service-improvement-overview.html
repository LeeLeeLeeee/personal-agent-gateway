30-second summary

새 기능보다 신뢰 기반이 먼저입니다

Chat과 Team Run은 실제 실행 경로가 연결돼 있지만, 인증 session과 Automation lifecycle, UI-runtime 계약은 제품이 약속하는 수준을 충족하지 못합니다.

ACTION REQUIRED
이야기 구조: 제품의 정체성과 실행 축 → 도메인과 실제 실행 흐름 → 끊긴 계약의 근거 → 개발·기획 개선안 → 하나의 실행 로드맵 순으로 읽습니다.
권장 실행 순서

유효 session 검증 → Worker/Scheduler lifecycle → background 오류 격리 → UI/runtime 계약 정합 → 핵심 E2E reliability gate

Application identity

제품과 도메인 지도

Personal Agent Gateway는 로컬 PC의 Codex·Claude CLI와 자동화 도구를 외부 브라우저에서 제어하는 1인용 control console입니다.

LOCAL-FIRST
Execution axis 01

Chat

한 Agent에게 세션 단위 작업을 맡기고 live activity, 승인, 중단, 결과를 관리합니다.

Execution axis 02

Team Run

Persona·Team·Rules snapshot으로 복합 목표를 계획하고 task별 실행과 최종 종합을 관리합니다.

Execution axis 03

Automation

Capability를 Job으로 실행하고 Schedule과 Artifact로 반복 결과를 관리합니다. 현재 lifecycle이 끊겨 있습니다.

시스템 컨텍스트

Chat과 Team Run은 UI → API → runtime → CLI/저장까지 연결됩니다. Automation은 Job/Schedule 생성 뒤 queue consumer와 due loop로 이어지는 app lifecycle이 빠져 있습니다.

State contracts

유지해야 할 실행 계약

  • Chat·Team 동일 ID의 중복 실행을 차단합니다.
  • Team Run은 Persona·Rules·model을 시작 시점 snapshot으로 고정합니다.
  • 재시작 시 active Team은 interrupted로 수렴하고 사용자 Resume을 기다립니다.
  • workspace·artifact·document의 path escape를 차단합니다.
Data ownership

데이터와 삭제·복구

  • Transcript는 대화 재현, activity는 진행 기록, audit는 보안 근거로 분리합니다.
  • Session 삭제는 JSONL·activity를, Team Run 삭제는 관계 row·전용 workspace를 함께 제거합니다.
  • Job/Schedule은 SQLite에 남지만 실행 lifecycle이 현재 빠져 있습니다.
  • Artifact register는 Full Access 성격상 workspace 밖 readable file도 허용합니다.

Development PM

유지보수·신뢰성·성능 진단

새 아키텍처를 덧씌우기보다 현재 강점을 유지하면서 깨진 경계부터 복구합니다.

Strengths

유지할 기술 기반

  • Service와 model client 주입으로 fake 기반 상태 전이 테스트가 가능합니다.
  • Chat·Team memory registry가 동일 실행 중복 진입을 차단합니다.
  • Team stale 상태를 interrupted로 복구하고 사용자 Resume을 요구합니다.
  • Persona·Rules·model options를 Run snapshot으로 고정합니다.
  • Path containment, SQLite foreign key와 WAL을 적용했습니다.
  • Backend 338개 테스트와 frontend 화면 테스트가 존재합니다.
Technical direction

개선 원칙

  • 대규모 ORM 또는 frontend state library 교체는 지금 하지 않습니다.
  • 인증·lifecycle·계약을 먼저 고친 뒤 반복되는 경계만 추출합니다.
  • 구조 분리는 기존 test를 전후 행동 계약으로 삼아 작은 단계로 진행합니다.
  • 성능은 fixture와 p50/p95를 먼저 정한 뒤 pagination·index를 적용합니다.

흔한 오해를 먼저 제거

✗ 오해 테스트가 338개이므로 lifecycle도 검증됐을 것이다.
✓ 실제 Service test가 많아도 app composition/startup 단절은 별도 lifespan E2E가 없으면 놓칠 수 있습니다.
✗ 오해 JobWorker와 SchedulerLoop 클래스가 있으므로 자동화는 동작한다.
✓ 실제 App lifespan이 start/stop을 소유하고 liveness를 검증해야 제품 기능입니다.
✗ 오해 Rules의 REQUIRED는 OS 권한처럼 행동을 강제한다.
✓ 실제 REQUIRED는 prompt instruction priority이며 permission boundary와 다릅니다.
✗ 오해 UI에 Max workers가 있으므로 task가 병렬 실행된다.
✓ 실제 현재 runtime은 task를 순차 처리하므로 문구를 바꾸거나 bounded concurrency를 구현해야 합니다.

R0-2

Worker/Scheduler lifecycle

M

FastAPI lifespan start/stop, queued recovery, due claim, configured concurrency 정책.

완료 기준
  • Lifespan 안 alive, 종료 뒤 stopped
  • Job queued→running→terminal
  • Due schedule 단일 Job과 restart 정책 test
R0-3

Background failure isolation

S

Per-job try/except, failed event, loop liveness, cancellation 구분, redaction.

완료 기준
  • 실패 Job 뒤 다음 queued Job 정상 실행
  • Cancel/shutdown과 genuine failure 구분
  • Secret 없는 structured error

Runtime capability flag와 validation, 미지원 기능 hide/disable, 실제 동작을 설명하는 label.

North-star 후보

사용자 수동 복구 없이 검증 가능한 결과로 끝난 작업 비율

보조 지표

지표 정의 개선 목적
Work success rateTerminal success 또는 accepted partial success 비율실행 신뢰성
Time to first useful result시작부터 첫 assistant/task result체감 속도
Recovery success rateFailed/interrupted 후 Resume/Retry 성공 비율복구 UX
Schedule reliability예정 실행 중 허용 지연 안 시작·완료 비율무인 자동화 신뢰
Result inspection timeTerminal부터 artifact/document 확인까지결과 탐색성
Configuration correction rate시작 직후 취소·재생성·설정 변경 비율설정 이해도
Error recurrence같은 환경에서 동일 오류 분류 반복 비율안내 품질

Prompt·파일·command 내용은 수집하지 않고 ID·status·duration·category·count만 로컬 집계합니다.

Definition of Ready
  • 사용자 문제와 영향 도메인이 명확함
  • Source of truth와 lifecycle owner가 정해짐
  • Failure/cancel/restart 동작이 정의됨
  • 보안·데이터 경계와 UI 문구가 합의됨
  • Fake 검증과 실제 smoke 검증 방법이 있음
Definition of Done
  • API/runtime/UI가 같은 계약을 사용
  • 정상·오류·중단·재시작 test 존재
  • Observability와 사용자 복구 action 존재
  • 성능 상한 또는 pagination/retention 정책
  • 운영 문서와 registry 갱신

주간 backlog review

  1. Critical/High incident와 contract gap을 먼저 확인합니다.
  2. Local-only 지표와 실제 사용 기록으로 사용자 마찰을 확인합니다.
  3. 한 항목에 개발 완료 기준과 사용자 성공 기준을 함께 적습니다.
  4. 실행하지 않는 UI와 사용하지 않는 config가 생기지 않았는지 확인합니다.
  5. 완료 항목은 knowledge/flow/ADR로 승격합니다.
