# OperationsView Team Run Policy Metadata Analysis

## 요약

- Root: `frontend/src/components/organisms/OperationsView/index.jsx`
- Modes: `api-state`, `test`
- Verdict: `OperationItems`가 받은 item을 표시하는 read-only 경계이므로 Team Run row에만 두 번째 meta line을 추가하면 된다. action/deep-link/backup/error 흐름은 변경하지 않는다.

## API / 상태 추적

- component는 `data.items`를 그대로 `OperationItems`에 넘기며 API 호출/mapper를 소유하지 않는다.
- 모든 row의 기존 첫 meta line은 `domain · id`이다. 신규 두 번째 line은 `domain === "team_run" && execution_policy`일 때만 렌더해야 job/schedule 출력이 변하지 않는다.
- 표시 필드는 `execution_policy`, `policy_status`, `queue_count`, `active_cycle_id`, `next_run_at`, `pause_reason`; 날짜는 기존 `fmtDateTime`을 재사용한다.
- `queue_count || 0`과 `policy_status || "ready"` 기본값을 사용하며 null optional fields는 문구를 만들지 않는다.
- onOpen/onResume/onRetry callback과 busyKey는 기존 row action 경계이므로 policy metadata가 영향을 주면 안 된다.

### 조회/action 흐름과 dependencies

- Operations screen 진입 → `loadOperations` → `api.operations()` → Gateway `operations`/`operationsError`/`operationsLoading` state → `OperationsView.data`.
- emergency stop은 `useConfirm` 승인 후 `run("emergency-stop")`; Gateway는 API mutation 뒤 jobs/schedules/teamRuns/settings/operations를 갱신한다.
- resume intake/create backup/verify backup은 `run(key)`가 `busyKey`를 설정·해제하고 Gateway mutation 뒤 operations를 refresh한다.
- row Resume은 domain별 resume API 후 전체 operations domains를 refresh하고, Retry는 job retry 후 operations를 refresh한다. Open은 target screen/id state를 바꾼다.
- `apiErrorAction`은 error status를 relogin/refresh/retry UI로 매핑한다. `useConfirm`은 destructive stop 승인 provider, `StatusBadge`는 health/item 상태 shared leaf, `fmtDateTime`은 backup과 신규 next-run 시각 helper다.
- `busyKey`는 한 action의 버튼 중복 실행을 막고 `run()`의 `finally`에서 항상 해제된다. policy metadata는 read-only text라 이 state와 상호작용하지 않는다.

## 테스트 / Stories

Story 검색 결과는 없다. 기존 tests는 health/diagnostics, deep link/resume/retry, emergency stop/backup, error detail과 retry recovery를 다룬다. relogin action은 component 코드에는 있으나 직접 unit coverage가 없으므로 이번 metadata 변경과 별도 backlog다. 다음 RED가 필요하다.

1. AUTO paused failure item이 policy/status/queue/next/pause/cycle을 표시.
2. TRIGGERED ready item과 optional null field가 불필요한 NEXT/CYCLE/pause text를 만들지 않음.
3. non-team item에는 두 번째 policy meta line이 없음.
4. 기존 Open/Resume/Retry callback assertions 유지.

## 권장 후속 작업

- `OperationItems` 내부의 title/meta block에 조건부 line 하나만 추가하고 formatter/helper 추출은 하지 않는다.
- 사용자 제공 가능성이 있는 `pause_reason`은 text node로만 렌더하고 HTML 삽입을 사용하지 않는다.

## 스킬 핸드오프

- 신규 metadata는 기존 row render에서 단순 문자열만 파생하며 추가 fetch/effect를 만들지 않는다. 별도 memoization은 이번 변경 근거가 없다.
- 기존 `.operations-row-meta` 스타일을 재사용하므로 새 CSS를 추가하지 않는다.
