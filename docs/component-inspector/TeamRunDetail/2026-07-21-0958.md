## API / 상태 추적

- Backend detail payload는 `api.teamRunDetail`이 policy snake_case를 camelCase로 바꾸고 controller의 `teamRunDetail` state를 거쳐 component에 들어온다. component가 fetch하지 않는다.
- component action은 주입 callback → controller의 trigger/retry/continue/restart → `api.triggerTeamCycle`/`api.retryAutoCycle`/`api.continueAutoCycle`/`api.restartAutoSeries` → `Promise.all(teamRunDetail, teamRuns)` refresh 순서다.
- `cycles`는 sequence 내림차순으로 파생된다. TRIGGERED preview는 이 배열에서 처음 만나는 `completed | completed_with_failures` Cycle을 사용하면 최신 settled summary/id가 함께 유지된다.
- 신규 `cycleInstruction`은 form 초안이며 성공한 `onTriggerCycle({instruction, previous_cycle_id})`에서만 비운다.
- AUTO action은 `activeAutoSeries.id`를 retry/continue에 전달하고 completed series는 `onRestartAuto()`를 호출한다. 동시에 두 action group을 보이지 않도록 `policyStatus` 분기가 단일 소유자여야 한다.
- `paused_interrupted`는 기존 `onResume` 버튼을 재사용한다. `canAddWork`에 `lifecycle_mode !== "continuous"`를 추가해 공통 CycleRequest 큐 우회를 막는다.
- `useTeamRunController.handleTeamEvent`는 cycle request/started/settled 및 series paused/completed 이벤트에서 선택 detail과 run list를 갱신해야 한다. 기존 selected-id ref가 다른 Run 이벤트를 차단한다.

### Local state와 기존 callback 부작용

| state/callback | 역할 |
|---|---|
| `workInput`, `submitting`, `workDialogOpen` / `onAddWork` | STANDARD 추가 업무 초안, pending disable, 성공 시 dialog close |
| `resuming` / `onResume` | interrupted resume 중 버튼 중복 실행 방지 |
| `canceling` / `onCancel` | active Run stop 중 버튼 중복 실행 방지 |
| `retryingTaskId` / `onRetryTask` | 선택 failed task 단위 pending 표시 |
| `selectedTaskId` | task 상세/보고서 선택 |
| `previewDoc` / `onLoadDocument` | 문서 content를 비동기 읽고 preview 표시 |
| `activeTab` | activity/results/documents 표시 전환 |
| `DecisionRequestPanel` local answers/submitting / `onAnswerDecision` | pending 질의 batch와 제출 중 disable |
| 신규 `cycleInstruction` / policy callbacks | TRIGGERED 초안과 AUTO action pending; 기존 work state와 분리 |

`StatusBadge`는 Run/Cycle/agent 상태 shared leaf, `Button`은 submit/resume/cancel action shared atom, `TeamTaskCard`는 task/owner/documentCount 표시와 상세 선택 `onOpen` leaf, `DocumentPreview`는 로드한 문서 content 표시 leaf다. task retry와 document open은 선택 뒤 `TaskDetailDialog`가 소유한다. 신규 policy panel은 이 네 public prop 경계를 변경하지 않는다.

## 테스트 / Stories

Story 검색 결과는 없다. 기존 component tests는 header/lanes, add-work pending/eligibility, interrupted resume, task retry, documents/tabs, continuous Cycle newest-first/lineage, cancel, decision flow를 다룬다. `client.test.js`는 aggregate/legacy 동일 policy shape와 trigger/retry/continue/restart URL/body를 이미 검증하므로 신규 UI tests는 callback payload와 표시 상태에 집중한다. 다음 RED가 필요하다.

1. TRIGGERED 최신 settled summary와 preview Cycle id, 성공 시 instruction clear/실패 시 유지.
2. queue count/active request 표시와 empty summary 안전 처리.
3. AUTO progress/countdown 및 paused_failure Continue/Retry, completed Restart, action group 상호배타성.
4. continuous Run에서 Add work가 숨고 STANDARD regression은 유지.
5. paused_interrupted에서 기존 Resume만 노출.
6. Gateway SSE cycle/series event가 selected detail/list refresh를 일으키고 다른 Run은 무시한다.

## 권장 후속 작업

- 기존 Cycle section과 agent toolbar를 재배치하지 말고 그 사이에 policy panel만 추가한다.
- 상태별 label/action을 render-time 파생하고 effect로 복제하지 않는다.
- 모든 async button은 기존 submitting/resuming 패턴처럼 중복 클릭을 막고 callback boolean을 존중한다.

## 스킬 핸드오프

- 구현은 `vercel-react-best-practices`의 derived-state/no-effect와 event-handler 원칙을 적용한다.
- 스타일은 기존 전역 CSS 명명 패턴을 유지하며 Task 8에 명시된 class만 추가한다.
