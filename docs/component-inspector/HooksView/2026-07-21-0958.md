# HooksView Triggered Team Run Target Analysis

## 요약

- Root: `frontend/src/components/organisms/HooksView/index.jsx`
- Modes: `api-state`, `test`
- Verdict: `HookForm`의 파생 target 후보를 backend와 동일한 continuous + plan_and_execute + triggered predicate로 좁히면 된다. create payload와 기존 Persona 경로는 변경하지 않는다.

## API / 상태 추적

- `teamRuns`는 parent가 전달하고 `HookForm`이 렌더마다 후보를 파생한다. 별도 effect/state copy가 없다.
- 현재 `continuousRuns`는 lifecycle/run_mode만 검사해 AUTO도 select에 노출할 수 있다. backend는 이를 400으로 거부하므로 UI predicate에 `execution_policy === "triggered"`가 필요하다.
- `selectedTargetTeamRunId`는 현재 state id가 후보에 있으면 유지하고, 아니면 첫 후보로 fallback한다. 필터 이름만 `triggeredRuns`로 바꾸되 이 동작을 유지한다.
- target 후보가 없으면 TEAM RUN button disabled, empty guidance가 표시된다. Persona target create payload와 connection test에는 영향이 없어야 한다.
- 기존 저장 Hook의 `targetSummary`는 teamRuns 전체에서 label을 찾으므로 AUTO legacy target의 표시까지 숨기지 않는다.

### Parent/API 흐름과 local dependencies

| 입력/동작 | 실제 경로와 부작용 |
|---|---|
| `teamRuns` | Hooks screen 진입 → `api.teamRuns()` → Gateway state → HookForm 후보 |
| `onCreate` | `api.createHook` → `api.listHooks` refresh → toast |
| `onToggle` | `api.updateHook` → list refresh; backend enable 검증 실패는 toast |
| `onRunNow` | `api.runHookNow` → hooks refresh, drawer open 시 runs refresh → toast |
| `onDelete` | `useConfirm` 승인 → `api.deleteHook` → hooks refresh/drawer clear → toast |
| `onOpenRuns`/`onCloseRuns` | `api.listHookRuns`와 Gateway drawer state를 변경 |
| `onOpenTeamRun` | HookRunsDrawer → drawer close → selected Team Run id 설정 → teams screen 전환 및 nav close |
| `onTestConnection` | `api.testHookConnection`; HookForm의 `testing/testResult`가 pending/result를 소유 |

`StatusBadge`는 Hook/HookRun 상태 표현 shared leaf, `PersonaPicker`는 Persona id 선택 shared organism, `useConfirm`은 HookRow delete 확인 provider, `fmtDateTime`은 polling/run 시각 표현 helper다. 이번 후보 predicate 변경은 이 경계를 수정하지 않는다.

검증 소유자는 create=`HooksService`, enable=`HooksService`를 호출하는 API PATCH, execution=`HookRunner._run_team_cycle` 내부 inline predicate로 나뉜다. UI filter는 편의/사전 차단이며 backend 세 경계를 대체하지 않는다.

## 테스트 / Stories

Story 검색 결과는 없다. 기존 frontend tests는 create toggle, Persona payload, continuous Team Run payload, connection test, run drawer/lineage, toggle/run-now를 다룬다. Backend service/API tests는 AUTO create/enable 거부를, HookRunner test는 실행 직전 AUTO target 거부를 보장한다. 다음 RED가 필요하다.

1. STANDARD, continuous AUTO, continuous TRIGGERED를 함께 주고 select에는 TRIGGERED만 표시.
2. 후보가 AUTO뿐이면 TEAM RUN disabled와 새 TRIGGERED 안내 문구 표시.
3. TRIGGERED 선택 create payload의 `target_team_run_id` 유지.
4. Persona create와 기존 Hook row target summary 회귀 유지.

## 권장 후속 작업

- `continuousRuns`를 명세 그대로 `triggeredRuns`로 치환하고 다른 form state/payload를 건드리지 않는다.
- backend predicate를 frontend에 재구현하므로 조건을 한 줄씩 명시해 누락이 눈에 보이게 한다.

- 스타일 변경은 필요 없고 empty copy만 정책 명칭을 명시한다.
